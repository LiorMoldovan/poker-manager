**Learned**:

The alternative — wait for Lior to notice each problem and report it — would have spread the same work across 3-5 sessions with worse continuity. When the user invites a "review and improvements" pass, treat it as license to enumerate ALL fixable issues, present a prioritized list, and let the user pick.

## 2026-05-13 — Photo chip-counting tip/pipeline alignment + defensive same-color sum (v5.60.2)

**Asked**: Lior took a real test photo on his phone after the v5.60.1 push and snapped a screenshot of the `PhotoCaptureModal` instructions. "anything to fix in this message?" Then a probing follow-up: "so there is no need to arrange it in stacks of 5?" Then final go-ahead: "do what you have to do, just make sure you dont damage anything that already works, remember we are in production".

**Did**:
- **Initial copy audit** turned up two small grammar/clarity issues (`instructionStep3` future-tense `ימלאו`, `instructionStep4` floating noun-phrase `תוצאה אוטומטית`) and one camera-angle wording question (`instructionStep2` "מהצד" — could be misread as horizontal/eye-level). Reported all three.
- **The poker-convention question opened the real investigation**: Lior asked "do I need to stack in 5s?" The intuitive poker convention is stacks of 5 for hand-counting, so users instinctively reach for that arrangement. Grep'd through `geminiAI.ts`, `chipGeometry.ts`, and `applyPhotoResult` in `ChipEntryScreen.tsx` to map the actual pipeline expectation. Found: (a) per-stack detection in `stackDetection.ts` finds vertical regions and maps each 1:1 to a chip color via HSL distance to the chip-selfie palette; (b) `MAX_REASONABLE_COUNT = 30` in `chipGeometry.ts` line 141 sets the per-stack ceiling at ~25; (c) `applyPhotoResult` in `ChipEntryScreen.tsx` line 528-534 iterates `result.stacks` and calls `updateChipCount(playerId, stack.chipId, stack.count, 'photo')` — OVERWRITE semantics, not additive. Conclusion: if a user splits one color into multiple piles, the chip-grid keeps only the LAST pile's count, silently undercounting by 50% or more.
- **Then found the smoking gun** while checking the secondary tips screen (the `💡 איך לשפר את הדיוק` link below the main instructions): `chips.photo.tip.maxStack` says, verbatim, *"גובה: עד 10 ז'יטונים בערימה. אם יש יותר — חלקו לכמה ערימות באותו צבע. ערימות גבוהות הן הסיבה העיקרית לטעויות."* — written for the v5.49 multi-shot consensus era when multiple same-color stacks WERE summed by the pipeline, but never updated when v5.59 changed the contract. Users following the tip's explicit advice were actively breaking their own counts.
- **Fixed the tip in HE and EN** to align with the new pipeline: "one stack per color, any height (up to ~25 chips), a single stack is more accurate than several small ones." Removed the "split if >10" advice entirely.
- **Defensive `applyPhotoResult` fix in `ChipEntryScreen.tsx`**: added a `summedByChipId: Map<string, number>` aggregation pass between `result.stacks` iteration and `updateChipCount` writes. Happy path (one stack per color) is bit-identical because each chipId appears exactly once and the sum equals the single stack's count. Multi-stack-same-color case now sums correctly instead of overwriting. `userEditedFields` skip moved into the aggregated loop to preserve the user-typed-value protection per chipId. Heavy comment added explaining why this matters and what scenarios it covers.
- **Did NOT touch the test card flow in `SettingsScreen.tsx`** (line 4440 has the same iteration pattern). Decision: the test card surface submits per-stack feedback rows to `chip_count_feedback`, and changing its `photoTestActualCounts` Map population semantics without rethinking the row submission shape was deemed higher production risk than the existing minor display oddity. Test card is debug-only and rarely hits the multi-stack-same-color case anyway.
- **Applied two small Hebrew copy polishes** at Lior's request: `instructionStep3` from `'משטח נקי, ערימות הקופה ימלאו את הפריים'` (future tense) → `'משטח נקי, מלאו את הפריים בערימות הקופה'` (imperative, matches the `מיינו` / `צלמו` pattern of the other steps); `instructionStep4` from `'תוצאה אוטומטית — תוכלו לערוך...'` (noun phrase) → `'הספירה אוטומטית — תוכלו לערוך...'` (sentence).
- **Skipped (at Lior's request)** the `instructionStep1` "left-to-right ascending" RTL friction concern (the AI pipeline expects this ordering, the visualization right below shows `1 2 3 4 5 6` LTR which makes it self-consistent), and skipped adding an explicit "don't split into stacks of 5" sentence (the visualization shows one chip per color, which already implies "one per color" enough).
- **Validated**: `npx tsc --noEmit` clean, `ReadLints` clean on both touched files. Mental-traced all 6 reachable scenarios (one stack per color happy path / multi-stack same color / user typed before photo / photo error early return / empty stack / re-photo over existing result) and confirmed happy path is bit-identical.
- **Bumped to v5.60.2** with 3 short user-facing bullets. Pushed to `origin/main` on top of v5.60.1.

**Learned**:
- **A tip screen is part of the contract.** When a pipeline's behavior assumption changes, you have to grep not just the code that uses the assumption but every place that DOCUMENTS it — user-facing copy, in-app help, tooltips, README, marketing text. The `maxStack` tip dated from May 2026 (v5.49.0) and was correct for that pipeline; the v5.59 rebuild in 2026-05-10 changed the contract but the tip stayed, silently misleading users for 3 days. Add a checkbox to the rebuild-a-pipeline ritual: "grep all user-facing strings that mention the old behavior".
- **"Defensive aggregation at the write boundary" is a cheap insurance pattern.** The `summedByChipId` Map cost 8 lines of code and a comment, and protects against arbitrary stack-detection oddities (multiple same-color piles, future pipeline changes, accidentally-introduced duplicates). The downside is zero — happy path produces bit-identical output. Always prefer "make the write idempotent against weird inputs" over "trust the producer not to send weird inputs", especially when the producer is an LLM and the writer is per-chipId state.
- **Trust the user when they push back on copy.** Lior's "I don't think there's a need for clear sentence not to do stack of 5" was right — the visualization below step 1 already shows one chip per color, and explicit "don't do X" sentences age badly (they preserve the wrong instinct in the user's memory). When in doubt about copy, ask, and take the answer.
- **The user's "I want safety" prompt ("don't damage anything that already works, remember we are in production") is the right cue to scope DOWN, not to add belt-and-suspenders.** I considered patching the test card flow too — same iteration pattern, same theoretical bug. But the test card has per-stack feedback row semantics that submitting summed values would silently shift, and the test card is debug-only. Patched only the production-critical path. Smaller PR = lower production risk = exactly what the user asked for.
