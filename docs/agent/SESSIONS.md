## 2026-05-13 — Delete Group actually deletes the group + multi-agent merge (v5.60.9, bundling queued v5.60.7 + v5.60.8)

**Asked**: Lior clicked "🗑️ מחק קבוצה" on the Group Management tab. Confirmation modal asked him to type the group name; he did; clicked confirm; modal closed; nothing happened. No toast, no deletion. "after confirming it didnt delete and didnt give any explanation why the group was not deleted". Then "now you have access to mcp, please do proper analysis and solve it". Then "please merge everything to git and bump version" — pulling in the queued v5.60.7 / v5.60.8 sibling-agent work.

**Did**:
- **Reproduced the failure live via MCP**: ran `DELETE FROM groups WHERE id = '<test-group>'` directly against the test group `ניסיון` (1 game, 3 players). Got `23503: update or delete on table "players" violates foreign key constraint "game_players_player_id_fkey" on table "game_players"`. Group untouched. The previous chat's `handleDeleteGroup` was CALLING the RPC correctly, and the RPC was returning the error correctly — but the error toast renders at the top of `GroupManagementTab` while the button sits at the bottom of a long settings page. Lior never saw it. That's the secondary UX bug; the primary functional bug is the FK 23503.
- **Two-bug root cause**:
  - (1) **PG cascade-from-groups doesn't satisfy the `game_players.player_id → players` NO-ACTION FK**. The original `delete_group` was just `DELETE FROM groups WHERE id = p_group_id`. PG fans out cascades to `games`, `players`, `group_members`, `settings`, etc. — order is implementation-dependent. When `groups → players` fires before `games → game_players` is fully drained, the FK check on each `players` row sees orphan `game_players` rows and aborts the entire transaction. The NO-ACTION FK exists for a reason: it's the DB-side safety net for the UI's `playerHasGames()` guard against deleting a player with games. So we can't change it to CASCADE — would silently destroy historical roster rows on individual-player delete.
  - (2) **Even with manual ordering, the bulk-delete + completed-game guards (043/050/051) would block the cleanup**. Migration 051 fixed cascade-detection for `game_players` (parents-all-gone check) but explicitly stated `games`/`players` are top-level with "no inbound FKs" — overlooking the inbound FK from `groups`. So a hand-orchestrated `DELETE FROM games WHERE group_id = X` on a group with 2+ games would hit `Bulk DELETE on games blocked: N rows`. And `block_completed_game_player_delete` (migration 050) would refuse direct deletes of completed-game `game_players` since they'd have `pg_trigger_depth() = 1` outside cascade context.
- **Migration 076 — flag-gated escape**: introduced a transaction-local setting `app.cascade_group_delete = '1'`, set only by the SECURITY DEFINER `delete_group` RPC after the owner-verification check. Both guard functions (`block_bulk_direct_delete` + `block_completed_game_player_delete`) now early-return when the flag is set. `delete_group` body rewritten to do an ordered cleanup: `DELETE game_players WHERE game_id IN (group's games)` → `DELETE games WHERE group_id = X` → `DELETE players WHERE group_id = X` → `DELETE groups WHERE id = X`. The flag is `is_local => true` so it auto-clears at COMMIT/ROLLBACK and can't leak across connections or outlast the RPC. Other callers (stale clients, direct DB writes from misbehaving code, etc.) still hit the guards exactly as before.
- **Sandbox-verified** end-to-end on Lior's `ניסיון` group inside an outer DO block with inner exception trap (so trigger function replacements + DELETEs roll back, but the captured variables make it back to the audit log inserted by the OUTER catch handler). Pre: 1 game / 3 players / 1 member / 1 settings. Post: 0 / 0 / 0 / 0. Three regression cases against `Poker Night` (239 games / 42 players) with the flag NOT set, all blocked correctly: bulk DELETE on games → "bulk blocked on games: 239 rows"; bulk DELETE on players → FK 23503 (caught upstream of the guard, still blocked); single-row DELETE on a completed-game `game_players` → "completed blocked: <uuid>". Applied 076 to live DB via `apply_migration` and confirmed all three function bodies now contain the flag check via `pg_get_functiondef`.
- **UX patch in `GroupManagementTab.tsx`**: added `modalError` + `modalBusy` state. `handleDeleteGroup` / `handleLeaveGroup` no longer close the modal on error — they set `modalError` and return. New inline error banner renders inside the modal above the action buttons (red, RTL-aware). Confirm + cancel + close-X all disabled while `modalBusy === true`; overlay-click is also gated. Other handlers (remove member, transfer ownership, regenerate invite) unchanged — their triggers are mid-page where the top-of-page toast IS visible.
- **Multi-agent merge bundling**: working tree had queued `v5.60.7` + `v5.60.8` entries in `version.ts` (friendly-notice copy polish across the no-AI-key empty states — summary/forecast/insights/comic/tts/photo/training, owner-vs-member variants) but never pushed (`origin/main` was at v5.60.6). Same shape as the v5.60.5 multi-agent merge: bumped to `v5.60.9`, added my three-bullet entry on top, left v5.60.7 + v5.60.8 entries untouched. Single commit, single push.
- **Validation**: `npx tsc --noEmit` clean (exit 0, ~38s full tree). `ReadLints` clean on `GroupManagementTab.tsx` + `version.ts` + `076-fix-delete-group-cascade.sql`. Live-DB verification via MCP: `pg_get_functiondef` for all three functions confirms `app.cascade_group_delete` is present. Group `ניסיון` still intact (1/3/1) — sandbox tests all rolled back.

**Learned**:
- **PG FK cascade order ≠ FK dependency order.** When `DELETE FROM parent WHERE id = X` has cascade rules to multiple children, and one child has a NO-ACTION FK to another, the order PG fires cascades is implementation-dependent and frequently DOES NOT respect the inter-child dependency. You can't trust "the cascades will sort themselves out" — write a sandbox test for any multi-child cascade scenario. The cheap version is `DELETE FROM <parent>` against a real row in a sandbox transaction and read the SQLERRM. Generalizes: NO-ACTION FKs combined with cascade-from-shared-grandparent is a foot-gun. Either CASCADE the FK (if the semantics allow), or orchestrate the deletion manually in dependency order.
- **A migration's "no inbound FKs" claim has a half-life.** Migration 051's logic was correct for the cascade chain `games → game_players`, but its prose generalization ("games and players are top-level with no inbound FKs") was wrong for the chain `groups → players → game_players` that hadn't been considered. The exact assertion the author makes about reachability is the thing that drifts when new tables/FKs are added. Future agents: when revisiting a migration's logic for a new failure mode, re-derive the FK graph rather than trusting the original migration's prose. The graph is one `pg_constraint` query away.
- **Toast-at-top-of-long-page is invisible UX for bottom-of-page actions.** This is the second time the "destructive action returned an error but the user never saw it" symptom shows up in this codebase (LESSONS.md 2026-05-07 was about silent EmailJS failures in the proxy; this one is the client-side mirror). The general fix is to render the error WHERE the user clicked, not WHERE the component decided to put the toast. Adopted: modal-bottom action errors render inside the modal, not in the page-top toast. Worth promoting to a UI rule if it shows up a third time.
- **Sandbox patterns: outer-block-catches-inner-rollback is the only way to get audit info out when the test must mutate data.** TEMP table inserts inside the savepoint also roll back. The pattern: declare local variables in the outer DO block, do the mutation inside an inner `BEGIN ... EXCEPTION WHEN OTHERS THEN ... END`, capture results into the outer variables before the deliberate rollback `RAISE EXCEPTION`, then the OUTER catch handler writes the captured variables to the audit table that the final `SELECT` can read. Six iterations to get this right today — saving the recipe.

- No outstanding follow-ups from this patch. The `Poker Night` production group is untouched; only `delete_group` callers benefit. Existing guard semantics for every other code path are byte-identical.
- Promotion candidate for LESSONS → rules: "render destructive-action errors in the modal, not in a top-of-page toast" if it surfaces a third time. For now, lesson lives in LESSONS.md.
